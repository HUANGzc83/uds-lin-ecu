📋 1. N_PCI (Protocol Control Information) 编码

N_PCI 字节是 LIN 传输协议的核心控制字段，位于每个诊断帧的第二个字节（NAD 之后）。 该字节分为 3 位类型字段 (bits 7:5) 和 5 位长度/序列字段 (bits 4:0)。

位 (Bits)字段 (Field)类型值宏定义帧类型说明
7 : 5
(Type)		 帧类型
(Frame Type)		 0 0 0 (0x00)LIN_PCI_SF Single Frame (SF) 数据在一帧内完成传输（≤6 字节）
0 0 1 (0x20)LIN_PCI_FF First Frame (FF) 多帧传输的第一帧，携带总长度
0 1 0 (0x40)LIN_PCI_CF Consecutive Frame (CF) 后续数据帧，携带序列号
0 1 1 (0x60)LIN_PCI_FC Flow Control (FC) 接收端发送的流控制帧
4 : 0
(Data)		 长度 / 序列
(Length / Seq)		 SF/FF: 数据长度 (0–6 SF, 0–15 FF high nibble)；CF: 序列号 (1–15, roll over)；FC: 流状态 (0–2) + BS + STmin
PCI 掩码宏定义:
LIN_PCI_MASK = 0xE0 — 提取 bits 7:5 帧类型字段
LIN_PCI_LEN_MASK = 0x1F — 提取 bits 4:0 长度/序列字段
提取示例: pci_type = pci_byte & LIN_PCI_MASK (保留高 3 位)；pci_len = pci_byte & LIN_PCI_LEN_MASK (保留低 5 位)

PCI 类型识别判断

PCI 字节值类型判断条件含义
0x00–0x06SF(pci & 0xE0) == 0x00单帧，低 5 位 = 数据长度 (0-6)
0x10–0x1FFF(pci & 0xE0) == 0x20首帧，低 4 位 = 总长度高 nibble
0x20–0x2FCF(pci & 0xE0) == 0x40续帧，低 5 位 = 序列号 (1-15)
0x30–0x3FFC(pci & 0xE0) == 0x60流控，低 5 位 = 流状态 (0-2)

📐 2. 帧布局 (Frame Layout) — LIN 诊断帧 = 固定 8 字节

每个 LIN 诊断帧的数据场固定为 8 字节 (ISO 17987-3)。第一字节为 NAD (Node Address for Diagnostic)， 第二字节为 PCI (Protocol Control Information)，剩余 6 字节根据帧类型存放 UDS 数据或控制参数。

🔹 Consecutive Frame (CF) — 连续帧

跟随 FF 之后的数据帧。PCI bits 7:5 = 2 (0x40), bits 4:0 = 序列号 (1–15, 循环)。后跟 6 字节 UDS 数据。

Byte 0Byte 1Byte 2Byte 3Byte 4Byte 5Byte 6Byte 7
NAD
0x01		 PCI
0x40|seq		 UDS 数据 (最多 6 字节) — 最后一帧可少于 6 字节
序列号: seq ∈ [1, 15], 从 1 开始递增，15 之后回绕到 0 (实际从 1 重新开始)
示例: CF1: PCI = 0x41, CF2: PCI = 0x42, ... CF15: PCI = 0x4F, CF16: PCI = 0x40

🔹 Flow Control (FC) — 流控制帧

接收端在收到 FF 后发送 FC 帧控制传输节奏。PCI bits 7:5 = 3 (0x60), bits 4:0 = 流状态 (Flow Status)。Byte 2 = BS (Block Size), Byte 3 = STmin。

Byte 0Byte 1Byte 2Byte 3Byte 4Byte 5Byte 6Byte 7
NAD
0x01		 PCI
0x60|FS		 BS STmin 未使用 (填充 0xFF)

Flow Status (FS) 取值

FS 值宏定义含义发送端行为
0x00LIN_FC_CONTINUE继续发送 (Continue To Send)按 BS 和 STmin 参数继续发送 CF
0x01LIN_FC_WAIT等待 (Wait)暂停发送，等待下一个 FC
0x02LIN_FC_OVERFLOW溢出 (Overflow)中止传输，接收端缓冲区已满
BS (Block Size): 每次允许连续发送的 CF 帧数。0 = 无限制 (发送完所有 CF 无需等待)
STmin (Separation Time minimum): 连续 CF 帧之间的最小时间间隔（见下方 STmin 编码表）
默认值: BS = 0 (无限制), STmin = 10 (10ms)

发送一条 20 字节 的 UDS 消息，通过 1 个 FF + 3 个 CF 完成传输。 序列号从 1 开始递增。Frame ID: 请求 = 0x3C, 响应 = 0x3D。

传输序列总结

步骤帧类型PCI 字节携带数据字节累计数据方向
1FF0x20 + len_lo=0x14Byte 0–4 (5 bytes)5/20Tester → ECU
FCFC0x60BS=0, STmin=10ms—ECU → Tester
2CF0x41 (seq=1)Byte 5–10 (6 bytes)11/20Tester → ECU

关键点:
• 序列号 从 1 开始，递增到 15 后回绕到 0（0x40 表示 CF seq=0）
• 最后一帧不足 6 字节时填充 0xFF 至 8 字节
• FC 帧由 接收端 (ECU) 主动发出，测试端收到后才能继续发送 CF
• STmin = 10ms 表示每两个 CF 之间至少间隔 10ms
• 总传输时间 ≈ 1×FF + 1×FC (N_Bs) + 3×CF × STmin ≈ 150ms + 3×10ms ≈ 180ms (理论最小值)

📡 LIN 帧序与 UART 串口协议

LIN 主从帧时序 — 从 Break 域到 Checksum，每个字节如何通过 UART 串口发送

📐 1. 完整的 LIN 帧结构 (ISO 17987-3)

LIN 总线上的一个完整帧由主节点发送帧头 (Header)，从节点回复响应 (Response)。 诊断帧使用固定的 ID：0x3C（主请求）和 0x3D（从响应）。

域长度内容发送方
Break≥13 bit times 低电平 强制拉低总线——所有从节点识别为新帧开始 Master
Sync1 字节 = 0x55 同步字节——从节点用它校准波特率 Master
PID1 字节 Protected Identifier — 6位ID + 2位奇偶校验。诊断: 0x3C(主请求) / 0x3D(从响应) Master
以上为 帧头 (Header) —— 始终由 Master 发送
Data0–8 字节 数据场。诊断帧固定 8 字节
Master 发 PID=0x3C → Master 接着发 UDS 请求数据
Master 发 PID=0x3D → Slave 接着发 UDS 响应数据 		Master (请求)
Slave (响应)
Checksum1 字节 校验和——增强型 (PID+Data) 或经典型 (仅 Data) 同 Data 发送方
🔑 PID 格式: 1 字节 = 6 位 Frame ID (bit 0–5) + 2 位奇偶校验 (bit 6–7)
bit 0–5 = Frame ID (0–63)，bit 6 = P0 = ID0⊕ID1⊕ID2⊕ID4 (偶校验)
bit 7 = P1 = ¬(ID1⊕ID3⊕ID4⊕ID5) (奇校验)
诊断帧 ID: 0x3C (ID=60) 主请求帧, 0x3D (ID=61) 从响应帧
✅ Checksum 格式:
增强型 (Enhanced, LIN 2.x): Checksum = ¬( sum(PID + Data0 + ... + DataN) )，包含 PID
经典型 (Classic, LIN 1.x): Checksum = ¬( sum(Data0 + ... + DataN) )，仅数据字节
诊断帧 (ID 0x3C/0x3D) 使用 增强型 校验。加法带进位回卷 (carry-around)。

🔌 2. 每个字节如何通过 UART 发送

数据场的每个字节不是"裸发"的——UART 串口会把每个 byte 包成 10 bit 的帧（Start + 8 bit + Stop）。 LIN 物理层把 UART 的 TX 信号连到 LIN 收发器芯片，再由收发器驱动单线总线。

📡 UART 串口帧 — 每个字节 = 10 bit 时长
Start = 0（拉低），8 个数据位 LSB 先发，Stop = 1（拉高）
Start
0		 BIT0 BIT1 BIT2 BIT3 BIT4 BIT5 BIT6 BIT7 Stop
1
绿 = Start 蓝 = 数据位 BIT0–BIT7 红 = Stop 9600/19200 bps — 1 字节 ≈ 0.52 ms
示例: 发送字节 0x35 (二进制 0011 0101)
内存中 MSB→LSB: 0 0 1 1 0 1 0 1
线 上 LSB→MSB: 0(Start) 1 0 1 0 1 1 0 0 1(Stop)
BIT0=1 最先, BIT7=0 最后 — 内存顺序恰好颠倒
在 LIN 19200 bps 下，1 字节 ≈ 0.52 ms

⏱️ 3. 主从帧调度时序

LIN 是主从架构——所有通信由 Master 发起。从节点只有在收到匹配 PID 后才回复。

📤 诊断请求帧 (PID=0x3C)
Master 发送全部 — Header 和 8 字节诊断请求都在同一帧内
Break Sync PID 0x3C NAD PCI UDS 请求 (6B) Chk
📥 诊断响应帧 (PID=0x3D)
Master 发 Header，Slave 接着填 8 字节诊断响应数据
Break Sync PID 0x3D NAD PCI UDS 响应 (6B) Chk
蓝 = Master 发送 黄 = Slave 回复 两帧间隔由 Master 调度表控制 (Slave P2 处理时间)
🔢 诊断帧 Header 之后:
如果 PID = 0x3C (主请求帧): Master 接着发 8 字节诊断请求
如果 PID = 0x3D (从响应帧): 指定 NAD 的 Slave 接着发 8 字节诊断响应

⚠️ 关键: 诊断请求和响应使用不同的帧——主请求 (0x3C) 之后下一帧是从响应 (0x3D)。 Master 在调度表里安排这两个帧的间隔，给 Slave 留出处理时间 (P2)。

📍 寻址模式与响应机制

物理寻址 / 功能寻址 / RCRRP 0x78 / SPRMIB / 响应 SID 计算

📡 物理寻址 vs 功能寻址

定义在 inc/uds/uds_core.h:205-209 — uds_addressing_t 枚举类型，控制 UDS 消息的寻址模式和 NRC 响应行为。

🔵 物理寻址 (UDS_PHYSICAL = 0)
  • ✅ 点对点通信 — 单个 ECU 响应
  • ✅ 所有 NRC 都发送给客户端
  • ✅ 正响应正常发送
  • ✅ uds_msg_t.ta_type = UDS_PHYSICAL
🟡 功能寻址 (UDS_FUNCTIONAL = 1)
  • ✅ 广播 — 多个 ECU 同时接收
  • 🚫 NRC 0x11 / 0x12 / 0x7E / 0x7F 被抑制
  • 🚫 正响应被抑制（默认行为）
  • ✅ uds_msg_t.ta_type = UDS_FUNCTIONAL
📋 功能寻址下被抑制的 NRC（ISO 14229-1 Tables 4-7）
NRC助记符名称原因
0x11SNSserviceNotSupported避免多 ECU 同时拒绝
0x12SFNSsubFunctionNotSupported避免多 ECU 同时拒绝
0x7ESFNSIAS子功能当前会话不支持功能寻址不需要区分会话
0x7FSNSIAS服务当前会话不支持功能寻址不需要区分会话
0x31ROORrequestOutOfRange广播请求不应因参数范围错误而失败
0x33SADsecurityAccessDenied功能寻址不应暴露安全状态
注意：功能寻址仍会发送其他 NRC（如 0x10, 0x13, 0x21, 0x22 等），用于指示真正的执行错误； 只抑制上表中的 "服务级" 拒绝类 NRC，以避免总线冲突。

⏳ RCRRP 机制 (0x78 — Request Correctly Received, Response Pending)

当服务器在 P2 时间内无法完成请求处理时，先发送 NRC 0x78 告知客户端 "请求已正确接收，响应待发送"， 随后在完成处理后发送最终响应。定义于 inc/uds/uds_core.h:159 和 inc/uds/uds_service.h:106-110。

客户端 ──── 请求消息 ────▶
│ 服务器接收，开始处理...
│ ⏱️ 处理时间 > P2
│ ◀── 7F XX 78 ──── NRC 0x78 (RCRRP)
│ 服务器继续处理...
│ ◀── SID+0x40 ... ──── 最终响应
📊 RCRRP 状态枚举 (uds_rcrrp_state_t)
值枚举含义
0RCRRP_NONE未发送 RCRRP
1RCRRP_SENT已发送 RCRRP (超时触发)
2RCRRP_PENDING_HANDLER处理程序执行中，等待判定
⚙️ 配置常量
// inc/uds/uds_service.h:64-66
#ifndef UDS_RCRRP_DELAY_MS
#define UDS_RCRRP_DELAY_MS 0u
#endif
默认值 0 = 处理前立即发送 RCRRP。
设置为非零值 > 0 表示延迟 N ms 后如未完成则发送。
💡 典型用法：-DUDS_RCRRP_DELAY_MS=50 — 处理时间超过 50ms 后自动发送 0x78，让客户端知道 "正在处理"。
🔗 RCRRP 响应消息格式
负响应格式
7F XX 78
最终正响应格式
SID+0x40 ...

🚫 SPRMIB — 抑制正响应指示位 (suppressPosRspMsgIndicationBit)

定义在 inc/uds/uds_core.h:191-197 — uds_subfunction_t 位段结构。子功能字节的 bit 7 设置为 1 时， 服务器不发送正响应消息（仅抑制正响应，负响应仍正常发送）。

📐 位段结构
typedef struct {
uint8_t value : 7; // bits 6:0 — 子功能值
uint8_t suppress_rsp : 1; // bit 7 = SPRMIB
} uds_subfunction_t;
🎯 示例对比
请求字节bit7子功能值行为
3E 0000x00✅ 发送正响应 7E 00
3E 8010x00🚫 抑制正响应
💡 关键：bit7=1 时的子功能值为 sf & 0x7F（去除 bit7），即 0x80 → 0x00，两个请求调用的服务处理函数完全相同。
📋 SPRMIB 使用场景
  • TesterPresent (0x3E) 在功能寻址中：使用 3E 80 抑制正响应，避免多个 ECU 同时回复
  • 周期性服务：减少总线负载，只关心负响应（仅在出错时反馈）
  • 广播控制：如关闭所有 ECU 的 DTC 设置 (0x85)，不需要每个 ECU 单独确认

🔢 响应 SID 计算

定义在 inc/uds/uds_core.h:255 — 请求 SID 与响应 SID 满足固定的偏移量关系。

正响应 SID = 请求 SID + 0x40
#define SID_TO_RESPONSE(sid) ((uint8_t)((sid) + 0x40))
服务名称请求 SID正响应 SID计算
DiagnosticSessionControl0x100x500x10 + 0x40 = 0x50
ReadDataByIdentifier0x220x620x22 + 0x40 = 0x62
TesterPresent0x3E0x7E0x3E + 0x40 = 0x7E
SecurityAccess0x270x670x27 + 0x40 = 0x67
SecuredDataTransmission0x840xC40x84 + 0x40 = 0xC4
RoutineControl0x310x710x31 + 0x40 = 0x71
🔴 负响应 SID 始终为 0x7F
7F + 请求SID + NRC
例：7F 22 7F = ReadDataByIdentifier 在当前会话不支持
🔍 反向计算宏
// 从响应 SID 还原请求 SID
#define SID_FROM_RESPONSE(sid)
((uint8_t)((sid) - 0x40))
例：0x62 → 0x62 - 0x40 = 0x22 (RDBI)
⚠️ 注意：请求 SID 范围限制在 0x10-0x3E 和 0x84-0x87（见 IS_REQUEST_SID() 宏）。对应的正响应 SID 为 0x50-0x7E 和 0xC4-0xC7。 任何消息的第一个字节如果不是 0x7F（负响应）且减去 0x40 后落在请求 SID 范围内，即为正响应消息。
